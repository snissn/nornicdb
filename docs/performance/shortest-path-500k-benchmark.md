# shortestPath benchmark — 500K nodes / ~3.97M edges

Single-process traversal latency at scale. Generated by:

```
go test ./pkg/cypher/ -run TestLargeScaleShortestPath_HopBuckets \
    -timeout 30m -largescale -v
```

Hardware: Apple M3 Max. Storage chain: BadgerDB → AsyncEngine → NamespacedEngine → StorageExecutor (the same chain the HTTP server uses).

## Setup

- 500,000 `Star` nodes across 1,000 sectors (500 per sector)
- 3,969,418 `HYPERLANE` edges (forward+reverse), avg ~16 per node
- Direct `BulkCreateNodes` / `BulkCreateEdges`, property index built **after** bulk insert + count verify (mirrors a typical "load → index → query" workflow)
- Reference BFS over 200 random sources to bucket pairs by exact hop distance
- 30 random pairs benched per depth bucket

## Seed timings

| phase                                     | time    |
| ----------------------------------------- | ------- |
| build nodes in memory                     | 215ms   |
| `BulkCreateNodes` (500K)                  | 234ms   |
| build edges in memory                     | 1.25s   |
| `BulkCreateEdges` (3.97M)                 | 1.54s   |
| flush AsyncEngine → Badger                | 3m28s   |
| count verify (500K nodes, 3.97M edges)    | 5µs     |
| build property index (post-flush)         | 157ms   |
| build reference adjacency in test process | 1.39s   |
| sample depth buckets (200 sources)        | 39ms    |
| **total seed wall-clock**                 | **3m32s** |

The 3m28s flush is the dominant seed cost — Badger's level-0 / level-1 compaction work for 4M edge writes. Once on disk, queries are read-only and benefit from the per-engine edge body cache and per-node adjacency cache.

## Observations

Three distinct latency regimes show up clearly:

1. **Depths 1–4: sub-millisecond.** Median 95µs–340µs. BFS visits a few hundred nodes; everything fits in the adjacency cache and the edge body cache.
2. **Depths 5–8: 1–3ms median, jumpy tail.** Mid-range traversals where the working set just barely exceeds what was cached from earlier benches; tail latencies (p95/max) are noisy because individual queries hit cold subgraphs.
3. **Depths 9+: linear scaling with hop count, ~14ms per hop through depth 39, then a 2× discontinuity at depth 40.** The median jumps from 282ms → 613ms.

### The depth-40 discontinuity

The cliff is consistent with cache-budget exhaustion. The default `nodeCacheMaxEntries` is 50,000. At depth ~40 over a graph of avg degree 16, BFS visits roughly 40 × 5K avg breadth ≈ 200K nodes — well past the 50K node-cache budget. Beyond that point every neighbor visit pays a fresh Badger iterator hit + edge body decode rather than serving from cache.

Tunable via `BadgerEngineOptions.NodeCacheMaxEntries` if you want sub-100ms response at depth 60+; raise it past the working set size or split per-node and per-edge cache budgets so a long traversal doesn't evict the path it just walked.

### The `min` column is misleading

The bench's first iteration in each bucket warms the executor's `SmartQueryCache` (60s TTL). Subsequent iterations with the same `(cypher, params)` shape — which several queries within a bucket happen to share by coincidence of the random pair sampler — short-circuit through that cache, hence the consistent ~25µs floor across all depths. The **median** is the honest signal; **p95/max** show how cache misses look at scale.

### Practical takeaway

For a 500K-node, ~4M-edge graph, this delivers:
- **Sub-millisecond** shortestPath at depths 1–4
- **Single-digit milliseconds** at depths 5–8
- **Linear ~14ms-per-hop** through depth ~40
- **~1s** for paths spanning the full sector chain (depth 60)

All of this is in-process; HTTP/Bolt overhead adds the usual 1–2ms on top.

## Reproducing

```sh
go test ./pkg/cypher/ -run TestLargeScaleShortestPath_HopBuckets \
    -timeout 30m -largescale -v
```

The `-largescale` flag is required; without it the test skips. Depth ceiling is `LARGESCALE_MAX_DEPTH=60` by default — set it lower to spend less time on long-tail buckets while iterating.
